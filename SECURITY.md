# Security Policy — VIdevX

## Підтримувані версії

| Версія | Підтримка безпеки |
|--------|-------------------|
| `main` (latest) | ✅ Активно підтримується |
| Старі гілки | ❌ Не підтримуються |

---

## Повідомити про вразливість

**Не** відкривайте публічний Issue для проблем безпеки.

Якщо ви виявили вразливість:

1. Напишіть листа на **security@videvx.dev** (або відкрийте [GitHub Security Advisory](https://github.com/vizarce/videvx/security/advisories/new))
2. Вкажіть:
   - Опис вразливості
   - Кроки для відтворення
   - Потенційний вплив
   - Версію/гілку, де виявлено
3. Ми відповімо протягом **72 годин**

## Що є в межах відповідальності

| Компонент | Статус |
|-----------|--------|
| Вихідний код фронтенду (`src/`) | ✅ В межах відповідальності |
| Supabase конфігурація (RLS, auth) | ✅ В межах відповідальності |
| npm-залежності (відомі CVE) | ✅ В межах відповідальності |
| Piston API (сторонній сервіс) | ❌ Не в нашій відповідальності |

## Відомі ризики та пом'якшення

### Виконання коду у пісочниці

Код виконується через [Piston API](https://github.com/engineer-man/piston), який запускає код **у ізольованому контейнері** на стороні сервера. Браузер користувача не виконує довільний код.

- Не надсилайте конфіденційні дані як код у пісочниці
- Piston API є публічним сервісом — не гарантується конфіденційність відправленого коду

### Supabase anon key

`VITE_SUPABASE_ANON_KEY` є **публічним ключем** (призначений для браузера). Безпека даних забезпечується через **Row Level Security (RLS)** на рівні PostgreSQL.

- Ніколи не використовуйте `service_role` ключ у фронтенді
- Завжди вмикайте RLS на всіх таблицях Supabase

### .env файли

Файл `.env` доданий до `.gitignore` і **не повинен комітитись** у репозиторій.

## Процес виправлення

1. Ми підтверджуємо вразливість (≤ 72 год)
2. Розробляємо виправлення у приватній гілці
3. Публікуємо патч і повідомляємо репортера
4. Після виправлення — публічне розкриття (координоване)
